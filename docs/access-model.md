# Access Model

## Два уровня ролей

В проекте есть два независимых уровня:

1. `app role`
2. `project role`

Это принципиально разные вещи.

## App Role

`app role` живёт на `User` через `spatie/laravel-permission`.

Текущий набор:

| Role | Смысл |
|------|-------|
| `owner` | Полный контроль над приложением |
| `tech_lead` | Расширенный рабочий доступ |
| `manager` | Управление проектами и доступами на уровне продукта |
| `developer` | Базовый рабочий доступ |

`app role` отвечает за:

- вход в продукт
- видимость экранов
- разрешение на CRUD по сущностям приложения

## Project Role

`project role` живёт в `project_memberships`.

Текущий набор:

| Role | Смысл |
|------|-------|
| `owner` | Владелец проекта |
| `tech_lead` | Технический лидер проекта |
| `manager` | Менеджер проекта |
| `developer` | Исполнитель внутри проекта |

`project role` отвечает за:

- роль пользователя внутри конкретного проекта
- ownership и распределение ответственности в контексте проекта

## Главное правило

Нельзя смешивать:

- `app role` не заменяет `project role`
- `project role` не даёт автоматический доступ в продукт

Пример:

- пользователь может быть `developer` на уровне приложения
- и одновременно `owner` внутри конкретного проекта

или наоборот:

- пользователь может быть `owner` на уровне приложения
- но вообще не быть участником проекта

## Как это проявляется в UI

### Users

На странице `Users` задаётся:

- кто есть в системе
- какой у пользователя `app role`
- активен ли пользователь

### Projects

На странице проекта задаётся:

- какие пользователи входят в проект
- какой у каждого `project role`

## Ownership в сущностях

В owner-facing модели ответственные люди привязаны через `user_id`:

- `resources.owner_user_id`
- `access_grants.user_id`
- `access_grants.owner_user_id`
- `secrets.owner_user_id`

## Reveal и чувствительные операции

Даже если пользователь есть в системе:

- raw secret values не участвуют в поиске
- reveal — это отдельное действие
- reveal должен оставаться осознанной операцией
